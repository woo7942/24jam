## 2026-04-XX (2주차) — DB 설계 & 견적 요청

### DB 스키마
- `move_requests` 테이블: 견적 요청 (주소, 일정, 가구, 박스, status enum `request_status`)
- `bids` 테이블: 입찰 (request_id, driver_id, price, message, estimated_duration_min, status enum `bid_status`)
- `service_type` enum 추가 (`general`, `half_packing`, `full_packing`)
- 컬럼 `service_type`을 `move_requests`에 추가, `estimated_price_min/max`는 nullable로 변경

### bids RLS 정책 5종
1. Drivers can view own bids (SELECT)
2. Customers can view bids on their requests (SELECT)
3. Drivers can insert bids (INSERT, role=driver 검증)
4. Drivers can update own pending bids (UPDATE)
5. Customers can accept bids on their requests (UPDATE)

### 견적 요청 작성 페이지
- `(customer)/request/...` 다단계 폼
- 주소 입력, 서비스 타입, 가구 선택, 박스 수, 메모, 희망 일자/시간
- 작성 완료 후 `move_requests`에 INSERT, `bid_deadline` 자동 설정

## 2026-04-XX (3주차) — 입찰 시스템 & 매칭

### 기사 페이지 라우팅 정리
- 처음에 `(driver)` 라우트 그룹과 `driver` 폴더 혼동 → 폴더 구조 꼬임
- 결국 `src/app/driver/requests/page.tsx`로 정리 (route group 제거)
- VS Code에서 폴더 이동 안 돼서 PowerShell `Move-Item`으로 처리

### 기사 대시보드 (`/driver/requests`)
- 입찰 가능한 요청(`status=open` AND `bid_deadline > now()`)
- 내가 이미 입찰한 요청은 "입찰 완료" 뱃지
- 각 요청의 입찰 수 표시
- 헤더에 "요청 보기" 버튼 추가 (driver 역할만)

### 기사 입찰 페이지 (`/driver/requests/[id]`)
- 가격(원) 필수, 메시지·예상 작업시간 선택
- 입찰 등록 / 수정 모드 자동 전환
- 마감 시간 지나면 입찰 폼 숨김

### 고객 견적 상세 (`/my/requests/[id]`)
- 받은 입찰 가격순 정렬, 최저가 뱃지
- "이 기사님 선택하기" 버튼 → bid status `selected`, request status `matched`

### bid_status enum에 `selected` 추가
- 처음엔 `accepted`로 코딩했다가 enum에 없어서 에러 → `selected`로 통일

### 트러블슈팅
- **users 테이블 RLS 무한 재귀**: 고객이 기사 정보 조회하려고 RLS 정책 추가했더니 `useAuth`가 무한 재귀로 500 에러
- **해결**: SECURITY DEFINER 함수 `get_bids_for_my_request(uuid)` 생성. RLS 우회하면서 권한 체크 내장
- **varchar vs text 타입 불일치**: 함수 RETURN 타입을 `text`로 선언했더니 `users.name`(varchar(50))과 안 맞음 → `::text` 캐스팅 추가

## 2026-04-XX (4주차) — 매칭 & Realtime 알림

### 매칭 시 자동 처리
- 고객이 기사 선택 시:
  1. 선택한 입찰 → `selected`
  2. 같은 요청의 다른 pending 입찰들 → `rejected`
  3. `move_requests.status` → `matched`, `selected_bid_id` 저장

### Realtime 활성화
- Supabase Database → Replication에서 `bids`, `move_requests` 테이블 ON
- `ALTER PUBLICATION supabase_realtime ADD TABLE ...`로 SQL에서도 처리 가능

### 기사 매칭 완료 섹션
- `/driver/requests` 상단에 "매칭 완료된 요청" 섹션
- 고객 이름 + 전화번호 + 전화 버튼 (`tel:` 링크)
- DB 함수 `get_my_matched_requests()` (SECURITY DEFINER)로 안전하게 조회

### Realtime 토스트 알림
- 기사: 내 입찰 status 변화 감지
  - `selected` → 🎉 "새로운 매칭이 도착했어요!"
  - `rejected` → 😢 "다른 기사님이 선택되었어요"
  - `cancelled` → ⚠️ "고객이 요청을 취소했어요"
- 새 견적 요청 INSERT 감지 → "📦 새로운 견적 요청이 도착했어요!"

### 헤더 매칭 뱃지 (1차)
- driver 역할에 한해 "요청 보기" 버튼 우상단에 빨간 원 + 매칭 카운트
- Realtime 구독으로 자동 갱신

## 2026-04-XX (5주차) — 취소·철회·삭제 & UI 개선

### 매칭 차단 로직
- 매칭된(`matched`) 요청은 다른 기사가 입찰 못 하도록 상세 페이지에서 분기
- 상태별 안내 화면: 매칭 완료 / 취소됨 / 마감 / 입찰 폼

### 고객 측 취소 기능
- **요청 취소**: `status=open`일 때만, 모든 pending 입찰 cancelled 처리 + 요청 cancelled
- **매칭 취소**: 이사 24시간 전까지만 가능. 선택된 입찰 cancelled + 요청 cancelled

### 기사 측 입찰 철회
- `status=pending`일 때만 가능
- bid status → `withdrawn` (enum에 이미 존재)
